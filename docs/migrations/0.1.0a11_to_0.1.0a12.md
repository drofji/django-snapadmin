# Migrating: 0.1.0a11 → 0.1.0a12

Two changes in `0.1.0a12` need manual action. There is **no database migration** and no model or API
schema change — both items are settings/wiring updates. The full notes for the release are in
[`docs/releases/0.1.0a12.txt`](../releases/0.1.0a12.txt); this guide covers only the steps you must take.

## 1. Celery task rename — update `CELERY_BEAT_SCHEDULE` and imports (BREAKING)

The background tasks moved from `snapadmin/api/tasks.py` (named `api.tasks.*`) to `snapadmin/tasks.py`
and are now namespaced `snapadmin.*`. This makes them discoverable by the stock
`app.autodiscover_tasks()` — previously a standard Celery setup never registered them, so Beat logged
`Received unregistered task` and the GDPR purge, error digest, database backups and async export
**silently never ran**. **There are no back-compat aliases** — the old names no longer resolve.

**Who is affected:** anyone running Celery (a worker and/or Beat) against SnapAdmin. If you don't run
Celery, there is nothing to do.

**Action — rename every task string in `CELERY_BEAT_SCHEDULE`:**

| Old (`api.tasks.*`) | New (`snapadmin.*`) |
|---------------------|---------------------|
| `api.tasks.purge_expired_tokens` | `snapadmin.purge_expired_tokens` |
| `api.tasks.purge_expired_data` | `snapadmin.purge_expired_data` |
| `api.tasks.send_error_digest` | `snapadmin.send_error_digest` |
| `api.tasks.run_export` | `snapadmin.run_export` |
| `api.tasks.run_db_backups` | `snapadmin.run_db_backups` |

```python
# settings.py — before
CELERY_BEAT_SCHEDULE = {
    "purge-expired-data": {"task": "api.tasks.purge_expired_data", "schedule": crontab(hour=3)},
}
# after
CELERY_BEAT_SCHEDULE = {
    "purge-expired-data": {"task": "snapadmin.purge_expired_data", "schedule": crontab(hour=3)},
}
```

If you imported any of these tasks directly in Python, update the import path too:

```python
# before
from snapadmin.api.tasks import run_export
# after
from snapadmin.tasks import run_export
```

If you set `CELERY_IMPORTS` explicitly to pull in `snapadmin.api.tasks`, drop it — the tasks are now
autodiscovered from `snapadmin/tasks.py`. After updating, restart the worker **and** Beat, and confirm
the tasks register (`celery -A yourproj inspect registered` should list the `snapadmin.*` names).

## 2. The system dashboard is now staff-gated by default

`DashboardView` used to render infrastructure details (hostname, processor, OS, database name, live
service health, `ALLOWED_HOSTS`) to **anonymous** callers. It now requires an authenticated staff user:
anonymous requests are redirected to the login page and authenticated non-staff users receive `403`.

**Who is affected:** anyone who wired `DashboardView` into a public URL and relied on it being reachable
without logging in (e.g. an external uptime check, or an intentionally public status page).

**Action — pick one:**

- Nothing, if the dashboard should be private (the new default). Reach it while logged in as staff.
- To keep it publicly reachable, opt out in settings:

  ```python
  SNAPADMIN_DASHBOARD_PUBLIC = True
  ```

  Only do this if the exposed infrastructure details are acceptable for anonymous viewers.

## Not a migration, but worth knowing

- **`nh3` is now a core dependency** (HTML sanitization for wysiwyg fields). `pip install` pulls it in
  automatically; no action needed. Rich-text values are sanitized before display — if a field's HTML is
  fully trusted, opt back into verbatim rendering with `SnapRichTextField(safe_html=True)`.
- **`SNAPADMIN_URL_PREFIX`** (new) lets you relocate the whole route surface if `/api/` collides with
  your project — additive, no action required. See the README.
- **`django-extra-settings` is now an optional extra**, not a core dependency (SnapAdmin's core never
  used it). If you use its dynamic `Setting` model and relied on SnapAdmin installing it transitively,
  install `django-snapadmin[extra-settings]` (or depend on `django-extra-settings` directly).
  Otherwise nothing to do — the base install is now one dependency lighter.
- **The wysiwyg editor (`django-ckeditor-5`) is now an optional `[wysiwyg]` extra** — it bundles
  CKEditor 5 (GPL/commercial), kept out of core so the base package stays permissively licensed. If you
  use `SnapRichTextField` / `wysiwyg=True`, install `django-snapadmin[wysiwyg]` and keep
  `django_ckeditor_5` in `INSTALLED_APPS`; otherwise you can remove `django_ckeditor_5` from
  `INSTALLED_APPS`. Using a wysiwyg field without the extra now raises a clear `ImproperlyConfigured`
  instead of an import error at startup.
