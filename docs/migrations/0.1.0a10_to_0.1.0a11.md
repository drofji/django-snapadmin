# Migrating: 0.1.0a10 → 0.1.0a11

**Breaking (pre-stable): migration history reset.** `snapadmin` and `demo` each carried five
incremental migrations accumulated since the first alpha (`0001`–`0006`). Both were squashed down to a
single `0001_initial.py` per app reflecting the current schema — no more incremental history to carry
into `0.1.0` stable. No model or API changes; this is purely a migration-file reorganization.

## Who needs to do something

Only installs that already ran `migrate` on `0.1.0a10` or earlier. A fresh install on `0.1.0a11+` needs
no action — `migrate` just applies the new `0001_initial.py` as normal.

## Steps

**Option A — drop and recreate the database** (simplest, if you can afford to lose local data):

```bash
dropdb yourdb && createdb yourdb
python manage.py migrate
```

**Option B — unwind migration state without dropping data:**

```bash
python manage.py migrate snapadmin zero
python manage.py migrate demo zero
# then re-apply the new squashed history:
python manage.py migrate
```

> The old and new `0001_initial` are **not compatible migration histories** — Django will not
> recognize `0.1.0a10`'s applied-migration rows as satisfying the new `0001_initial.py`. Skipping this
> step causes `migrate` to fail with an inconsistent migration history error.
