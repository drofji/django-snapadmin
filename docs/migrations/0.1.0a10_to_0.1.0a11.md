# Migrating: 0.1.0a10 → 0.1.0a11

**Breaking (pre-stable): migration history reset.** `snapadmin` and `demo` each carried five
incremental migrations accumulated since the first alpha (`0001`–`0006`). Both were squashed down to a
single `0001_initial.py` per app reflecting the current schema — no more incremental history to carry
into `0.1.0` stable. No model or API changes; this is purely a migration-file reorganization.

## Who needs to do something

Only installs that already ran `migrate` on `0.1.0a10` or earlier. A fresh install on `0.1.0a11+` needs
no action — `migrate` just applies the new `0001_initial.py` as normal.

## Steps

**Option A — drop and recreate the database** (simplest, if you can afford to lose local data):

```bash
dropdb yourdb && createdb yourdb
python manage.py migrate
```

**Option B — reset only the migration *records*, keep your data (non-destructive):**

The squash changed no tables — only the recorded migration *history* differs. So the fix is to drop
the stale history rows (`0001`–`0006`) for the two apps and mark the new `0001_initial` as already
applied, **without running any table DDL**:

```bash
# 1. Back up the database first.

# 2. Delete the stale migration records for both apps (this touches only the
#    django_migrations bookkeeping table, never your data tables).
python manage.py shell -c "from django.db.migrations.recorder import MigrationRecorder as R; R.Migration.objects.filter(app__in=['snapadmin','demo']).delete()"
#    …or in raw SQL:  DELETE FROM django_migrations WHERE app IN ('snapadmin', 'demo');

# 3. Record the new squashed 0001_initial as applied — no SQL is executed, the
#    existing tables are left exactly as they are.
python manage.py migrate snapadmin --fake
python manage.py migrate demo --fake
```

Verify with `python manage.py showmigrations snapadmin demo` (each should read `[X] 0001_initial`) and
`python manage.py migrate --check`.

> **Do not use `migrate snapadmin zero`.** Reversing to `zero` *drops the app's tables and their data*,
> and once the intermediate `0002`–`0006` files no longer exist Django can't walk that path anyway — it
> fails with an inconsistent / nonexistent-node history error. The record-reset above is both
> non-destructive and reliable.
>
> Why not `replaces=[…]`? A squashed migration that lists the old names in `replaces` would let Django
> reconcile the history automatically, but the intermediate migration files were removed in this
> release, and the squashed file keeps the name `0001_initial` (which would collide with the
> `0001_initial` it needs to replace). Retrofitting `replaces` now would risk fresh installs already on
> `0.1.0a11`, so the supported upgrade path is the manual record-reset above.
